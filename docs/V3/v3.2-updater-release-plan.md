# RicercaCasa V3.2 — updater-first installazione e release pubblica

Data: 14 luglio 2026

## 1. Obiettivo

La prima release pubblica `v3.0.0` deve essere realmente installabile da un computer che non possiede la repository sorgente.

L'updater deve essere operativo già dalla prima release e deve gestire:

- prima installazione;
- creazione della struttura locale;
- generazione e uso dei secret;
- download e verifica degli asset;
- pull delle immagini Docker;
- avvio PostgreSQL;
- migrazioni database;
- avvio backend e frontend;
- verifica health e readiness;
- wizard iniziale;
- stato persistente dell'installazione;
- aggiornamenti successivi;
- backup prima degli aggiornamenti;
- rollback delle immagini quando possibile.

La prima installazione e gli aggiornamenti devono usare la stessa state machine e lo stesso motore operativo.

Non deve esistere un installer completo separato che esegue direttamente migrazioni e avvio dell'applicazione aggirando l'updater.

---

## 2. Esperienza utente finale

L'utente deve avere soltanto:

- Docker Engine oppure Docker Desktop;
- Docker Compose v2;
- `curl` oppure uno strumento equivalente;
- una shell POSIX compatibile per il bootstrap Linux/macOS.

Installazione prevista:

```bash
curl -fsSL https://github.com/joxDev12/ricercaCasa/releases/download/v3.0.0/install.sh | bash
```

Il bootstrap deve:

1. verificare Docker e Docker Compose;
2. creare `~/.ricercacasa`;
3. scaricare e verificare gli asset minimi di bootstrap;
4. generare i secret locali con permessi `600`;
5. avviare esclusivamente l'updater bootstrap;
6. mostrare l'URL locale dell'updater;
7. non avviare direttamente database, migrate, backend o frontend.

L'utente apre:

```text
http://127.0.0.1:8081
```

Da quel momento l'updater esegue la prima installazione completa.

- riavvio dell'updater senza perdita del job corrente;
- installazione riprendibile dopo interruzione.

## 4. Separazione bootstrap e stack applicativo

### 4.1 Bootstrap Compose

File previsto:

```text
deployment/bootstrap-compose.yaml
```

Deve avviare soltanto l'updater.

Può contenere:

- immagine updater precisa;
- volume stato;
- volume installazione;
- volume backup;
- accesso controllato al Docker socket;
- porta locale updater;
- secret setup necessari al bootstrap.

Non deve avviare:

- PostgreSQL;
- migrate;
- backend;
- frontend.

### 4.2 Compose release

File previsto:

```text
deployment/compose.release.yaml
```

Deve:

- non contenere `build:`;
- usare esclusivamente `image:`;
- usare immagini GHCR versionate;
- non usare `latest` come riferimento operativo;
- mantenere i volumi nominati;
- mantenere il servizio one-shot `migrate`;
- mantenere healthcheck e dipendenze;
- non dipendere dalla repository sorgente;
- essere gestibile dall'updater.

Servizi:

1. `database`
2. `migrate`
3. `backend`
4. `frontend`
5. `updater`

Sequenza applicativa:

```text
database healthy
→ migrate exit 0
→ backend healthy
→ frontend healthy
→ installazione completata
```

## 5. Accesso dell'updater a Docker

Per gestire installazione e aggiornamenti, l'updater deve poter eseguire operazioni Docker strettamente definite.

Soluzione iniziale prevista:

```yaml
volumes:
  - /var/run/docker.sock:/var/run/docker.sock
  - ${RICERCACASA_HOME}:/installation
```

L'immagine updater deve includere:

- Docker CLI;
- Docker Compose plugin;
- strumenti di verifica checksum;
- strumenti HTTP necessari;
- `pg_dump` compatibile con PostgreSQL usato dalla piattaforma, oppure un container dedicato al backup.

### 5.1 Avvertenza di sicurezza

L'accesso al Docker socket equivale, di fatto, ad accesso amministrativo all'host Docker.

Per ridurre il rischio:

- updater esposto soltanto su `127.0.0.1`;
- nessuna API di shell arbitraria;
- nessun parametro utente trasformato direttamente in comando;
- allowlist rigida delle operazioni;
- argomenti passati tramite array, senza shell interpolation;
- manifest validato con schema;
- immagini consentite solo sotto namespace GHCR previsto;
- digest obbligatori;
- percorsi locali normalizzati e confinati in `~/.ricercacasa`;
- job concorrenti vietati;
- log senza secret;
- token bootstrap non restituito al browser;
- setup token usato solo tra updater e backend;
- eventuali endpoint mutativi protetti da token locale e controllo Origin/Host;
- documentazione esplicita dei privilegi richiesti.

In futuro valutare un socket proxy con allowlist, ma non è obbligatorio per il primo MVP se la superficie updater rimane minima e locale.

## 6. State machine unica

La stessa state machine deve gestire prima installazione e aggiornamenti.

Stati minimi:

```text
idle
preflight
fetching_manifest
verifying_manifest
downloading_assets
verifying_assets
pulling_images
starting_database
waiting_database
creating_backup
running_migrations
starting_backend
waiting_backend
starting_frontend
waiting_frontend
finalizing
completed
failed
rolling_back
rolled_back
```

Per la prima installazione:

- `creating_backup` viene saltato perché non esiste ancora un database precedente;
- il resto del flusso resta uguale.

Ogni job deve salvare almeno:

```json
{
  "jobId": "uuid",
  "kind": "install|update",
  "fromVersion": null,
  "toVersion": "3.0.0",
  "status": "running",
  "phase": "pulling_images",
  "startedAt": "ISO-8601",
  "updatedAt": "ISO-8601",
  "completedAt": null,
  "error": null,
  "rollback": null
}
```

Requisiti:

- scrittura atomica dello stato;
- ripresa dopo restart;
- un solo job attivo;
- identificazione delle fasi idempotenti;
- timeout per ogni fase;
- errori strutturati;
- log persistenti;
- nessun avanzamento se la verifica precedente fallisce.

## 7. Prima installazione gestita dall'updater

Flusso obbligatorio:

1. verifica che non esista un'installazione completata;
2. verifica Docker daemon;
3. verifica Docker Compose v2;
4. verifica spazio disco;
5. verifica permessi directory locale;
6. carica manifest release;
7. valida manifest con JSON Schema;
8. verifica firma o checksum degli asset;
9. verifica namespace e digest immagini;
10. scarica `compose.yaml`, `release.env` e asset richiesti;
11. scrive i file in modo atomico;
12. esegue `docker compose pull` con immagini precise;
13. avvia PostgreSQL;
14. attende database healthy;
15. esegue il servizio one-shot `migrate`;
16. verifica exit code `0`;
17. avvia backend;
18. attende `/health/ready`;
19. avvia frontend;
20. attende health frontend;
21. salva versione installata;
22. abilita wizard iniziale;
23. completa setup profilo tramite backend;
24. marca installazione completata;
25. mostra pulsante per aprire RicercaCasa.

Il bootstrap `install.sh` non deve eseguire direttamente i punti 12-24.

## 8. Migrazioni database dalla prima release

Il servizio `migrate` deve essere presente anche nella release `v3.0.0`.

Requisiti:

- usa la stessa immagine del backend;
- usa la versione esatta indicata nel manifest;
- attende PostgreSQL healthy;
- usa Docker secrets;
- esegue tutte le migrazioni pending;
- termina con exit `0` solo quando lo schema è allineato;
- il backend non parte se migrate fallisce;
- readiness backend rifiuta migrazioni mancanti o sconosciute;
- log migrate salvati dall'updater;
- nessuna migrazione eseguita dal browser;
- nessun downgrade automatico del database.

Prima installazione:

```text
database vuoto
→ migrate applica 001...017
→ db:status allineato
→ backend ready
```

## 9. Aggiornamenti successivi

Flusso minimo per `3.0.0 → 3.0.1`:

1. recupera manifest remoto;
2. confronta versione installata e disponibile;
3. mostra note rilascio;
4. richiede conferma utente;
5. esegue preflight;
6. verifica spazio;
7. blocca aggiornamenti concorrenti;
8. crea backup PostgreSQL;
9. verifica backup non vuoto;
10. scarica nuovi asset;
11. verifica checksum;
12. esegue pull immagini per digest;
13. aggiorna file release in modo atomico;
14. esegue migrate one-shot;
15. avvia backend nuovo;
16. attende readiness;
17. avvia frontend nuovo;
18. verifica health finale;
19. aggiorna versione installata;
20. conserva log e riferimento backup.

In caso di errore:

- non cancellare il backup;
- ripristinare i riferimenti alle immagini precedenti quando possibile;
- riavviare la versione precedente se lo schema database resta compatibile;
- non eseguire downgrade automatico delle migrazioni;
- mostrare errore e istruzioni di ripristino;
- salvare stato `failed` o `rolled_back`.

## 12. Checksum degli asset

Asset release previsti:

```text
install.sh
bootstrap-compose.yaml
compose.yaml
release.env.example
manifest.json
manifest.schema.json
checksums.txt
```

`checksums.txt` deve usare SHA256:

```text
<sha256>  install.sh
<sha256>  bootstrap-compose.yaml
<sha256>  compose.yaml
<sha256>  release.env.example
<sha256>  manifest.json
<sha256>  manifest.schema.json
```

Regole:

- checksum generati in CI dopo la creazione degli asset finali;
- installer verifica gli asset prima dell'uso;
- updater verifica gli asset prima di sostituire file locali;
- sostituzione atomica tramite file temporaneo e rename;
- fallimento immediato su mismatch;
- mai continuare con asset non verificato.

Valutare firma con attestazioni GitHub/Sigstore in una fase successiva; SHA256 è il requisito minimo iniziale.

## 21. Criteri per il merge della PR #6

La PR `v3/pre-deploy → main` può essere unita soltanto quando:

- updater gestisce realmente la prima installazione;
- installer bootstrap avvia soltanto updater;
- Compose release non contiene `build:`;
- immagini release sono costruibili;
- workflow release è presente;
- manifest reale è generabile e validabile;
- checksum sono generabili e verificabili;
- smoke release funziona senza repository;
- migrazioni partono dall'updater;
- backend/frontend vengono avviati dall'updater;
- stato installazione persiste;
- tutti i job CI sono verdi;
- revisione finale non trova blocchi;
- PR non è più Draft solo dopo queste verifiche.

La sola CI di sviluppo attuale non è sufficiente per dichiarare pronta la prima release.

## 25. Definizione di completato

La V3.2 è completata quando un utente su un computer pulito può eseguire un solo comando di bootstrap, aprire l'updater locale e ottenere una RicercaCasa `3.0.0` funzionante con database migrato, backend ready, frontend healthy e configurazione persistente, senza clonare la repository e senza costruire immagini localmente.
